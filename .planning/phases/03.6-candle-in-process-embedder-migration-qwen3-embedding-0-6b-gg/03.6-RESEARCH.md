## Summary

**Verdict: pivot away from the GGUF cheap path. Use safetensors + candle-transformers `models::qwen3::Model` (NOT `quantized_qwen3` and NOT `quantized::llama`).** Three findings flip the prior plan:

1. **`quantized_qwen3::ModelWeights::forward()` returns logits over vocab, not hidden states.** The cheap path locked in PROJECT.md ("`convert_hf_to_gguf.py` -> `candle-transformers quantized::llama` loader") cannot extract the last-token hidden state needed for last-token-pooling embeddings. Source: `candle-transformers/src/models/quantized_qwen3.rs` -- `forward()` body explicitly does `h.narrow(1, l-1, 1)` then applies `lm_head` projection before returning. [CITED: candle source via WebFetch]
2. **The official Qwen3-Embedding-0.6B-GGUF maintainer cautions the GGUF tokenizer is broken and llama.cpp is ~10x slower than ONNX on CPU for embedding.** Source: HF repo discussion #8. [CITED: huggingface.co/Qwen/Qwen3-Embedding-0.6B-GGUF/discussions/8]
3. **`candle-transformers::models::qwen3::Model::forward()` returns `(batch, seq_len, hidden_size)` -- exactly the hidden states we need.** Pair with safetensors download (1.19 GB, single file) + `tokenizers` crate for the BPE tokenizer.json (HF official, 11.4 MB). Standard candle pattern, no Python sidecar, no GGUF tokenizer bug. [CITED: candle source `qwen3.rs`]

**Top 3 risks:**
1. **Cosine equivalence with current ollama output is NOT guaranteed bit-equivalent.** ollama wraps the same Qwen3-Embedding-0.6B but ollama applies its own template/pooling that we do not control. §9.8 version-hash WILL flip. Plan accordingly: full reindex of poc.db + fsc.db is mandatory, not optional. Equivalence threshold should be cosine ≥ 0.97 (NOT 0.999) on the 30-query regression set; 0.999 will reject normal numerical drift between two valid implementations of the same model.
2. **Cold-start time on CPU for a 600M-param F32 safetensors load is likely 5-20s, not the spec's "≤30s" trivial-pass.** This affects the `index` command UX (already noticed for ollama because it's a daemon). New cost: every `query` CLI invocation pays this. Mitigation: run `serve` mode persistent so `query` hits the warm A2A endpoint, NOT the CLI cold path. Confirm in plan whether Cmd::Query CLI is now considered debug-only or must stay fast.
3. **Windows MSVC + candle compile time is the unknown landmine.** No GPU on this Win11 host, so CPU-only is the path. candle-core CPU on MSVC compiles cleanly per multiple references but adds ~30-90s to release builds. Acceptable but worth a 30-min cargo-build smoke before locking the deps.

**Recommended next step for planner:** Split into 2 plans (see §6 Recommended Plan Structure). Plan 1 = sourcing + loader (high uncertainty, must spike before committing to API design). Plan 2 = equivalence validation + cross-corpus eval (gated on Plan 1 producing working `embed()`).

### Pattern 1: lazy model load via OnceCell

**What:** load model + tokenizer once on first `embed()` call, cache thereafter.
**When to use:** in `Embedder::new()` is too eager for `Cmd::Query` cold-path; in every `embed_once()` call is wasteful for `Cmd::Index` loop.
**Example:**
```rust
// Source: standard candle pattern, see candle-examples/examples/qwen/main.rs
use std::sync::OnceLock;

struct EmbedderState {
    model: candle_transformers::models::qwen3::Model,
    tokenizer: tokenizers::Tokenizer,
    device: candle_core::Device,
}

pub struct Embedder {
    state: OnceLock<EmbedderState>,
}

impl Embedder {
    pub fn new() -> Self {
        Self { state: OnceLock::new() }
    }

    fn ensure_loaded(&self) -> Result<&EmbedderState> {
        if let Some(s) = self.state.get() { return Ok(s); }
        // First call: download (or read from cache) + load
        let device = candle_core::Device::Cpu; // ARCH §7 D-W5; CUDA/Metal opt-in feature flags
        let api = hf_hub::api::sync::Api::new()?;
        let repo = api.model("Qwen/Qwen3-Embedding-0.6B".to_string());
        let weights_path = repo.get("model.safetensors")?;
        let tokenizer_path = repo.get("tokenizer.json")?;
        let cfg_path = repo.get("config.json")?;

        let cfg: candle_transformers::models::qwen3::Config =
            serde_json::from_slice(&std::fs::read(cfg_path)?)?;
        let vb = unsafe {
            candle_nn::VarBuilder::from_mmaped_safetensors(
                &[weights_path], candle_core::DType::F32, &device,
            )?
        };
        let model = candle_transformers::models::qwen3::Model::new(&cfg, vb)?;
        let tokenizer = tokenizers::Tokenizer::from_file(tokenizer_path)
            .map_err(|e| anyhow::anyhow!("tokenizer load: {}", e))?;
        let _ = self.state.set(EmbedderState { model, tokenizer, device });
        Ok(self.state.get().unwrap())
    }
}
```

### Pattern 2: last-token pooling with left-padding awareness

**What:** standard last-token pool extracted from HF official Python pipeline.
**Source:** [HF Qwen3-Embedding-0.6B model card](https://huggingface.co/Qwen/Qwen3-Embedding-0.6B) (`last_token_pool` function).
**Example:**
```rust
// For batch_size=1 (typical embed_once call): just take the last real token.
// attention_mask sum = sequence_length; index = sum - 1.
// Tokenizer padding side = "left" per HF, so the last_real_token_idx == seq_len - 1
// in left-padded mode. We use right-padding-by-default for B=1 (no padding needed)
// so last_real_token_idx = input_ids.len() - 1.

let last_hidden = model.forward(&input_ids, 0)?;     // shape: (1, seq_len, 1024)
let (_b, seq, _h) = last_hidden.dims3()?;
let pooled = last_hidden.narrow(1, seq - 1, 1)?      // shape: (1, 1, 1024)
                        .squeeze(1)?                   // shape: (1, 1024)
                        .squeeze(0)?;                  // shape: (1024,)
// L2 normalize
let norm = pooled.sqr()?.sum_all()?.sqrt()?.to_scalar::<f32>()?;
let normalized: Vec<f32> = pooled.to_vec1::<f32>()?
    .into_iter().map(|x| x / norm.max(1e-12)).collect();
```

For B=1 we skip the attention-mask gymnastics. If batching is added in Phase 4 for throughput, port the full HF `last_token_pool` with left-padding mask handling.

### Pattern 3: query/passage prefix discrimination (already present)

The current `embedder.rs` `Role::Query` / `Role::Passage` enum stays unchanged. Source-of-truth strings:

```rust
// Verbatim from HF official model card (https://huggingface.co/Qwen/Qwen3-Embedding-0.6B):
// Query: "Instruct: {task_description}\nQuery:{query}"
// Passage: raw text, no prefix
//
// Current QUERY_INSTRUCT in embedder.rs:6 -- VERIFY VERBATIM PRESERVATION:
//   "Instruct: Given a natural language code search query, retrieve the most
//    relevant code symbol from a TypeScript codebase\nQuery: "
//
// Note the trailing "Query: " with space; HF official format is "Query:{query}"
// (NO space before {query}). The current ollama setup has a space. This is one
// of the things to verify: the old ollama and the new candle MUST use identical
// prefix strings or §9.8 hash diverges trivially.
```

**Action item for planner:** lock the canonical prefix string in code as a `const` and use it for BOTH the new candle path AND the equivalence-test ollama snapshot. Consider whether "Query: " (with space) vs "Query:" (no space) was a long-standing latent bug -- the HF docs say no space, current code has space. Decision: **keep the space** (match what's currently indexed in poc.db); document as a known deviation in the version-hash text.

### Anti-patterns to avoid

- **DO NOT** load the model inside `embed_once()` -- that re-loads on every call (catastrophic). OnceCell or OnceLock is mandatory.
- **DO NOT** recreate `tokenizers::Tokenizer` per call -- it's also expensive (~50ms).
- **DO NOT** ship F16 weights initially. F32 is the safe default for cosine-equivalence (reduces one source of numerical drift). If memory pressure shows up, F16 is a follow-up optimization with re-validation.
- **DO NOT** trust `quantized_qwen3` for embeddings without first checking if forward() can be made to output hidden states (it can't without forking).
- **DO NOT** silently change the QUERY_INSTRUCT string. Any change is a §9.8 version-hash bump = full reindex.

## Don't Hand-Roll

| Problem | Don't build | Use instead | Why |
|---------|-------------|-------------|-----|
| BPE tokenization for Qwen3 | Custom BPE state machine | `tokenizers` crate `Tokenizer::from_file("tokenizer.json")` | HF official tokenizer.json is 11.4 MB of vocab + merge rules; getting BPE right by hand is the original "expensive fallback" from PROJECT.md backlog. Don't. |
| safetensors mmap loading | Hand-parse safetensors | `candle_nn::VarBuilder::from_mmaped_safetensors` | Standard candle pattern, handles dtype + device transfer atomically |
| HuggingFace Hub download | Manual HTTPS + cache directory layout | `hf-hub` crate | Honors HF_HOME, integrates with HF cache, handles ETag/revisions |
| Qwen3 transformer forward pass | Custom attention + RMSNorm + RoPE | `candle_transformers::models::qwen3::Model::forward` | Already implemented and used in candle examples; covers GQA + RoPE correctly |
| Last-token pool | Custom | Pattern 2 above (4 lines using `narrow + squeeze`) | "Hand-roll" is fine but copy from HF official to avoid off-by-one |
| L2 normalization | Custom | Pattern 2 above (`sqr + sum_all + sqrt + map`) | Standard pattern, 3 lines |

**Key insight:** the original PROJECT.md "cheap path" (GGUF + `quantized::llama`) saved BPE-tokenizer + forward-pass implementation cost. Pivoting to safetensors RE-INTRODUCES the BPE-tokenizer cost (now solved by the `tokenizers` crate, which already ships from HF) and the forward-pass cost (already solved by `candle-transformers::models::qwen3`). **Net cost is roughly the same as the GGUF path would have been.** The "expensive fallback" framing in PROJECT.md was based on assuming the safetensors path required hand-writing those pieces. With `tokenizers + candle-transformers` it doesn't.

## Common Pitfalls

### Pitfall 1: Forgetting GGUF tokenizer caveat
**What goes wrong:** spend a day on the GGUF + `quantized_qwen3` path, then discover (a) forward returns logits not hidden states, (b) GGUF tokenizer is reported broken upstream.
**Why it happens:** PROJECT.md and ARCH §9.10 both encode the GGUF path as "cheap" without verifying the API surface. PROJECT.md was written before the `quantized_qwen3` source was inspected.
**How to avoid:** start with the safetensors path. The "expensive fallback" framing in PROJECT.md is wrong (see "Don't Hand-Roll" above).
**Warning signs:** `quantized_qwen3::ModelWeights::forward` ends with an `lm_head` matmul; if you see that in the source, hidden states are not exposed.

### Pitfall 2: F32 vs F16 numerical drift breaks cosine-equivalence
**What goes wrong:** load weights as F16 to save RAM, then cosine vs ollama drops to 0.93-0.95 mean -- below any reasonable equivalence threshold.
**Why it happens:** ollama defaults to F32 on CPU (verify), candle defaults to whatever DType you pass to VarBuilder. If they differ, accumulated round-off across 28 transformer layers diverges.
**How to avoid:** use `DType::F32` for the equivalence-validation milestone. Once equivalence is locked + reindex done, F16 can be re-evaluated as a separate measured optimization (with its own version-hash bump if it changes search results materially).
**Warning signs:** equivalence cosine consistently 0.92-0.95 with no input-dependent pattern -> dtype mismatch.

### Pitfall 3: Padding side mismatch
**What goes wrong:** HF official Python uses `padding_side='left'` for Qwen3-Embedding (decoder-only model with left-padding pooling). If you tokenize with `padding_side='right'` and then pool by `seq_len - 1`, the last token is `<pad>`, not the real semantic token. Embeddings come out garbage.
**Why it happens:** `tokenizers` crate defaults to right-padding.
**How to avoid:** for B=1 (single text per call -- the embed_once contract today), no padding is needed at all. Just pass raw input_ids of length `actual_seq_len`. For batched embedding (Phase 4), explicitly set `tokenizer.with_padding(Some(PaddingParams { direction: PaddingDirection::Left, .. }))`.
**Warning signs:** F1 embedding on a 5-token query has cosine ~0 with the same query embedded by ollama -> padding side bug.

### Pitfall 4: Trailing-space in QUERY_INSTRUCT
**What goes wrong:** current `embedder.rs:6` has `"Query: "` (trailing space). HF official has `"Query:"` (no trailing space, immediate concat with query text). Switching during migration changes the prefix and silently invalidates the §9.8 hash even if everything else is identical.
**Why it happens:** copy-paste drift; the current poc.db was indexed with the trailing space and beat 60% precision, so the trailing space is part of the empirical config.
**How to avoid:** preserve the existing `QUERY_INSTRUCT` constant byte-for-byte. The §9.8 version-hash text should record this deviation explicitly.
**Warning signs:** code review nitpicks "should be Query:{query} not Query: {query}" -- reject the change without an empirical re-evaluation.

### Pitfall 5: Cold-start invisible until query path
**What goes wrong:** `Cmd::Query` invocation feels broken because it sits 5-15s on first call; user keypresses or aborts.
**Why it happens:** OnceLock model load on first `embed()` blocks the main thread.
**How to avoid:** for `Cmd::Index`, this is fine -- one-time cost amortized over thousands of symbols. For `Cmd::Query` CLI, either (a) accept the cold-start as a known limitation (CLI is debug-tier; production query goes through `Cmd::Serve` warm A2A endpoint), or (b) add an `eprintln!("loading model...")` progress indication on first call.
**Warning signs:** "query is broken" reports that turn out to be cold-start.

### Pitfall 6: 2307-symbol full FSC reindex time inflation
**What goes wrong:** ollama could embed ~50 symbols/sec on this host (measured). candle CPU on a 600M-param F32 model embeds ~5-15 symbols/sec on a typical x86 CPU (1 batch=1, no GPU, no flash-attn). 2307 symbols / 10 sym/sec = 230s = ~4 min, or 2307 / 5 = 460s = 8 min worst case. NOT 17 min as ARCH §7.4 capacity envelope estimated for a "typical embedder" at 50 sym/sec.
**Why it happens:** ARCH §7.4 was written for the old Snowflake-arctic-embed-xs (22M params). Qwen3-Embedding-0.6B is 27x larger; per-call latency scales accordingly.
**How to avoid:** budget 5-15 min for FSC full reindex on this CPU host. NOT a blocker for Phase 03.6 acceptance (one-time cost). Consider batching as a Phase 4 throughput optimization.
**Warning signs:** indexer feels slow but is making steady progress; not the same failure mode as ollama hang.

### A3: Tokenize + forward + last-token pool + L2 normalize
```rust
// Source: HF Qwen3-Embedding-0.6B model card (Python last_token_pool ported)
//         candle-core docs.rs (Tensor::narrow, squeeze, sqr, sum_all)
let prompt = match role {
    Role::Query => format!("{}{}", QUERY_INSTRUCT, text),
    Role::Passage => text.to_string(),
};
let encoding = state.tokenizer.encode(prompt, true)
    .map_err(|e| anyhow::anyhow!("tokenize: {}", e))?;
let ids = encoding.get_ids().to_vec();
let input_ids = candle_core::Tensor::new(&ids[..], &state.device)?
    .unsqueeze(0)?;  // (1, seq_len)
let last_hidden = state.model.forward(&input_ids, 0)?;  // (1, seq_len, 1024)
let (_b, seq, _h) = last_hidden.dims3()?;
let pooled = last_hidden.narrow(1, seq - 1, 1)?
                        .squeeze(1)?
                        .squeeze(0)?;  // (1024,)
let pooled_f32: Vec<f32> = pooled.to_vec1()?;
// L2 normalize in pure Rust
let norm = pooled_f32.iter().map(|x| x * x).sum::<f32>().sqrt().max(1e-12);
let normalized: Vec<f32> = pooled_f32.into_iter().map(|x| x / norm).collect();
return Ok(normalized);
```
