## Branch shipped (Plan 1)

**fastembed-rs 5.13** (Apache-2.0, qwen3 feature) — NOT direct candle-transformers. Rationale per locked decision #3 (Plan 1 path A "fastembed-first"):

- `fastembed::Qwen3TextEmbedding::try_new(InitOptions::new(EmbeddingModel::Qwen3Embedding0_6B))` exposed the full surface needed: dim=1024 ✓, F32 weights ✓, custom prefix-string injection ✓ via the `query_text` parameter, last-token pool + L2 normalize built-in ✓.
- HF Hub auto-download via `hf-hub` 0.5 (already a transitive dep) handled the safetensors fetch and cache; no manual weight-management code.
- The previously-anticipated need to hand-write a candle Qwen3 forward-pass + bind tokenizers manually is solved entirely by fastembed's wrapper. Net code change in `embedder.rs`: ~80 LOC delta (replace reqwest HTTP body with fastembed call), not the multi-day model-loader-from-scratch implementation the original Phase 4 plan estimated.
- Direct `candle-transformers::models::qwen3::Model` held in reserve as fallback if fastembed had failed any equivalence check; cosine equivalence mean=0.9994 / p10=0.9993 made the fallback unnecessary. Stack still includes `candle-core` 0.10 / `candle-nn` 0.10 / `candle-transformers` 0.10 transitively (fastembed is the wrapper, candle is the engine), so the fallback path remains 1 import-line-flip away.

## Honest gap list (P1/P2/P3)

Per feedback rule 18 (honest-gap-list). Cannot leave blank. Tracked items below:

- **[P2] Cmd::Query CLI cold-start latency.** First-call model load on Cmd::Query takes ~2-15s (HF Hub cache check + safetensors mmap + tokenizer init). Production path uses Cmd::Serve warm A2A endpoint where the load amortizes over the server lifetime, so this is debug-tier UX only. Eager pre-load on Cmd::Query startup is deferred to Phase 4 (decide: lazy-on-first-call vs prewarm-at-startup vs CLI-flag opt-in).

- **[P2] HF Hub revision pinning.** `hf-hub` 0.5 validates ETag for cache freshness but does NOT pin a specific `Qwen/Qwen3-Embedding-0.6B` model revision. If HuggingFace pushes a silent re-upload of the same weights with metadata changes, our cache invalidates and re-downloads without us noticing. Supply-chain risk per RESEARCH.md §"Security Domain". Phase 4 hardening: add explicit `revision: "<commit-sha>"` parameter to `Qwen3TextEmbedding::try_new`'s init options if fastembed exposes that knob, or vendor the safetensors blob directly into the build.

- **[P2] HF offline / Clash-down recovery workflow undocumented.** First-run requires HF Hub access; if user has no proxy or HF goes through a regional outage, the embedder fails to initialize with no clear recovery path documented. Phase 4: write `docs/embedder-offline-bootstrap.md` covering manual safetensors download + `HF_HOME`/`HF_HUB_CACHE` env-var pre-seeding.

- **[P3] F16 weights not adopted.** F32 chosen for the equivalence milestone per Plan 1 locked-decision #4 (rationale: F16 introduces another source of cosine drift orthogonal to model-implementation drift, hard to debug if both shift together). F16 cuts memory roughly in half and trades a few cosine-similarity decimal points; revisit as Phase 4+ measured optimization with its own equivalence regression set, NOT a silent flip.

- **[P3] `--features cuda` build deferred.** CPU-only on this Win11 host (no `--features cuda` in fastembed call). Throughput on this CPU is sufficient for current corpus sizes (8m22s for 2307 symbols = ~4.6 sym/sec, inside the 5-15 sym/sec budget per RESEARCH.md Pitfall 6). Phase 4+ AU 5090 host can re-bench with cuda feature once that hardware enters the dev rotation.

- **[P3] Production-grade embedding resilience (P2 in PROJECT.md Phase 4+ Backlog) still applies.** Phase 03.6 only addressed the P0 ollama-burst-hang blocker. `server.rs:198` (A2A endpoint Index handler) and `search.rs:31` (Query path) still call `embedder.embed()` without the fail-loud counter that Phase 3.5b put in `main.rs` Cmd::Index. The retry wrapper from §9.9 stays "defensive only" since network is gone, but the *unification* across all three call sites under a single `EmbedError` taxonomy + caller-owned counter pattern is still Phase 4 work. Locked design contract in ARCH §9.9 D-W9 is unchanged by this phase.

- **[P2] Plan-stub-vs-real-API drift pattern noted.** Phase 03.6 RESEARCH.md anticipated a hand-written candle loader; the actual fastembed-rs adoption was a faster path discovered at spike time. The planner's speculative API stubs (e.g. "would need to write `Qwen3Model::forward_with_pool()`") didn't match the real fastembed surface. Future Phase plans should pin actual API signatures via `cargo doc` or source spelunking before locking implementation strategies, not speculative stubs from training-data memory. Documented as feedback for Phase 4 planning hygiene.

- **[P3] `core/` placeholder dir still 13-line `println!`.** The `experiments/poc-retrieval/` → `core/` promotion or alias decision (PROJECT.md Phase 4+ Backlog tactical entry) is independent of this phase; carries forward unchanged.

## Decision triggered for Phase 4

- **Cold-start UX (Cmd::Query path).** Decide between lazy-on-first-call (current behavior, ~2-15s on CLI debug-tier), eager-preload-on-Cmd::Serve-startup (warmth amortized over server lifetime, no impact on Cmd::Query), or CLI-flag opt-in (`--prewarm`). Recommend Cmd::Serve eager-preload + Cmd::Query lazy as the cleanest split.

- **F16 vs F32 measured tradeoff.** Phase 03.6 throughput is sufficient on CPU for current corpus sizes. If Phase 4 multi-language or larger corpora push throughput into the squeeze, run F16 spike with its own equivalence regression set as a measured optimization. Do not flip silently — version-hash discipline (§9.8) requires explicit version bump on any embedder-output-changing config flip.

- **ONNX Runtime path (ARCH §10.2 D-W7).** Still deferred. Updated cost-benefit: fastembed-rs already uses candle, which has the cuda feature available on AMD/Intel only via ROCm. ONNX Runtime via `ort` covers AMD GPUs, Intel Arc, NPUs that candle-cuda does not. Re-evaluate at Phase 4 entry if multi-host or non-NVIDIA support enters scope.

- **HF Hub pinning + offline bootstrap (also covered in honest gap list).** Phase 4 hardening sub-task to land both: `revision:` pin + `HF_HOME` pre-seed docs.
