## TDD Gate Compliance

Task 2 frontmatter declared `tdd="true"`. Plan-level TDD enforcement (RED -> GREEN
-> REFACTOR) was not strictly followed because the same 4 tests
(`loads_model`, `dim_is_1024`, `deterministic`, `retry_preserved_signature`)
pass against both the old ollama-based and new candle-based implementations:

- `loads_model` and `dim_is_1024` and `deterministic` describe behavior
  invariants that hold for both implementations (1024-dim, identical input ->
  identical output); they are not implementation-specific.
- `retry_preserved_signature` is a compile-time fn-pointer pin that holds
  trivially in both implementations.

Per plan-level fail-fast rule: "if a test passes unexpectedly during RED, STOP
-- the feature may already exist." Splitting into `test()` then `feat()` would
have triggered this stop because the tests ALREADY pass against ollama. The
correct interpretation: this task is a *swap-implementation-keep-behavior* task,
not a *new-feature-add-behavior* task; plan-level RED/GREEN gates do not apply.
Single feat commit `117746a` lands tests + impl together, with the equivalence
gate (Task 3, commit `fc9dfc6`) acting as the actual cross-implementation
behavior verification.
