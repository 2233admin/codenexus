## TL;DR

Split monolithic `graph_build.rs` (574 LoC, 4 concerns inline) into a
pure-fn pipeline: `edge-find` (per-language **lang_extractor** consuming
**RepoCtx**) -> `edge-resolve` (**CallResolver** over **AliasIndex** +
**SymbolIndex**) -> `edge-store` (writes **ResolvedEdge**). Lift sentrux
plugin/{manifest,profile,loader} + parser/{imports,ast_import_walker}
verbatim (~2200 LoC). Cherry-pick path-alias logic from sentrux
resolver/suffix.rs (~400 LoC). Build **CallResolver** native (~300 LoC).
Add `[calls]` capability to plugin.toml schema. Ship `parser` sub-crate.
Move `Imports` out of `edges` table to a separate `alias_decls` table
(A3.3). T1-T7 must stay green; REQ-10 B1-B7 in `[65.9%, 69.9%]`.

## Risks

- **R1 (high):** Sentrux `parser/imports.rs` + `ast_import_walker.rs`
  use sentrux types (`Symbol`, `EntryPoint`, `LanguageProfile`) that
  codenexus lacks. Type-adaptation pattern from 04.5-02a applies but
  fan-out is larger here (~885 LoC vs 478 LoC). Estimate +30% over
  raw lift cost.
- **R2 (medium):** Cherry-pick from `resolver/suffix.rs` is the
  riskiest slice -- entangled with sentrux's structural-analysis
  logic. Could underestimate. Mitigation: cherry-pick is W2 and
  isolated from W1's verbatim lift, so over-run only delays W2-W5
  not W0-W1.
- **R3 (medium):** REQ-10 G-D gate depends on a clean candle eval
  run, which is currently flaky on this machine due to HF cache I/O
  conflict (os error 183, see progress.txt 2026-05-02 entry). Not
  04.5-03's fault, but blocks closure. Workaround: run G-D on
  pre-cached cache (offline-bootstrap doc).
- **R4 (low):** `[calls]` TOML schema design -- need a proto plugin.toml
  for TS during W1 to validate the schema before downstream waves
  consume it. Mitigation: schema design is part of W1 deliverable, not
  a hidden dependency.
- **R5 (low):** fsc.db reindex (~8m22s) cost on every dev iteration
  validating G-E. Mitigation: keep poc.db as the regression target
  (smaller, faster); fsc.db only at G-D / G-E.

## Unresolved Questions

- **UQ1: `parser` sub-crate vs `core` module?** 04.5-01 promoted to
  workspace; 04.5-02a created `metrics` sub-crate (precedent). But
  `parser` is consumed by `core::graph_build` directly -- circular dep
  if `parser` re-exports `Symbol` (defined in core today). Resolution
  options: (a) move `Symbol` to a `types` sub-crate consumed by both;
  (b) define `Symbol` in `parser` and re-export from `core`. Decide
  during W1 plan.
- **UQ2: schema migration auto vs reindex-required?** Current G-E pick
  is reindex-required with loud error. Alternative: SQL-level migration
  preserving `Imports` rows in `alias_decls` (lossy mapping, since
  symbol-level Imports doesn't map cleanly to `(file, alias,
  target_file, target_member)`). Lean toward reindex (clean), but the
  lossy migration may save user time. Validate during W0.
- **UQ3: minimum viable `[calls]` TOML schema** -- Codex flagged this
  has to be designed in 04.5-03, not 04.5-07, but the actual fields
  are open. Probably: `call_expression_kinds: [string]` (e.g.
  `["call_expression", "new_expression"]` for TS), `method_call_pattern:
  string` (regex or AST shape for `obj.method()`), `dynamic_dispatch:
  bool`. Validate against TS + Python during W4.
- **UQ4: tree-sitter-graph adoption timing** -- L6 punts to Phase 4+
  but worth cheap probe (≤2hr) at end of 04.5-03 to lock the decision
  for 04.5-07. Alternative: defer to Phase 4 first slice.
- **UQ5: confidence per-language calibration** -- L5 makes it possible
  but doesn't lock when. Likely 04.5-07 when Python lands and we have
  a comparison surface. Confirm during 04.5-07 plan.
