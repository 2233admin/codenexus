<objective>
Fix the Makefile/Cargo build chain. `Makefile:4` declares
`CORE_BIN := codenexus-core` and `Makefile:25` does
`cp .../target/release/$(CORE_BIN)$(...).exe ...`, but
`experiments/poc-retrieval/Cargo.toml:46` declares
`[[bin]] name = "poc-retrieval"`. Built artifact is
`poc-retrieval.exe`, not `codenexus-core.exe`. `make build-server`
fails at the `cp` step BEFORE Plan 04-03's E2E harness can run.

This plan lands ONE direction: rename the `[[bin]]` `name` from
`"poc-retrieval"` to `"codenexus-core"` (option (a) per REVIEWS.md
HIGH#2). Preserves `package.name = "poc-retrieval"` (crate identity
unchanged; only the binary artifact name changes). The `poc-retrieval`
name was always meant to be temporary pending Phase 4 productization
— this slice is the productization moment for the binary surface.

Purpose: unblock Plan 04-03 E2E harness which invokes
`./bin/codenexus` (the Go wrapper that //go:embeds the Rust core
binary) AND/OR `target/release/codenexus-core.exe` directly.

Output:
- `experiments/poc-retrieval/Cargo.toml` line 46 changed:
  `name = "poc-retrieval"` -> `name = "codenexus-core"`.
- `cargo build --release` produces `target/release/codenexus-core.exe`
  (Windows) / `codenexus-core` (Unix).
- `make build-server` cp step succeeds because
  `experiments/poc-retrieval/target/release/codenexus-core.exe` now
  exists at the path Makefile expects.
</objective>
