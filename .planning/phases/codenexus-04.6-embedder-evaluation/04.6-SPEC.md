**Headline finding:** Qwen3-Embedding-0.6B is currently SOTA on MTEB-Code among
**open-weight Apache 2.0 self-hostable** models. The only verified-better option
is its own 8B sibling (+5.27 pp at 13x VRAM cost). All "code-specialized" 2026
candidates (Qodo, CodeXEmbed, Voyage-code-3) either lag on score, lock behind
non-permissive licenses, or require fastembed-rs adapter rewrite.

## Hypothesis

**H1 (load-bearing):** swapping Qwen3-Embedding-0.6B → Qwen3-Embedding-8B lifts
B1-B7 precision_at_5 by >= 3 pp (from 67.9% baseline) on the existing 30-query
suite, while keeping CPU-only inference latency under 10 min for full poc.db
re-index.

**H2 (alt path):** a 2026-Q1/Q2 model NOT in current shortlist (e.g. an unreleased
Microsoft / Google / Meta drop landing between 2026-05-03 and SPEC ratification)
beats Qwen3-Embedding-8B on B1-B7 by >= 2 pp.

**H3 (null):** no candidate beats current 0.6B on B1-B7 by >= 3 pp once measured
on the actual CodeNexus eval suite (MTEB-Code rankings don't transfer).

## Scope (in)

- B1-B7 +/-2pp re-baseline of current Qwen3-Embedding-0.6B (control)
- Same-eval shootout against:
  - Qwen3-Embedding-8B (zero-effort swap, primary candidate)
  - Microsoft Harrier-oss-v1-27B (if base architecture confirmed candle-compatible)
  - One closed API control (Voyage-code-3) for upper-bound calibration

## Eval framework

**Reused from Phase 03.6:**
- `eval/queries.json` -- 30 queries (A1-A10 broad / B1-B10 narrow / C1-C10 negative)
- `eval/embed_equivalence_30q.json` -- per-query expected top-5 ground truth
- `precision_at_5` per query, mean over 30 queries
- B1-B7 narrow-band reproducibility gate: |new - 67.9%| > 2pp = winner OR loser

**Per-candidate metrics:**

| Metric | Definition | Threshold for "winner" |
|--------|------------|-----------------------|
| B1-B7 precision_at_5 (mean) | Phase 03.6 reused | >= 70.9% (current 67.9% + 3pp) |
| Full-suite precision_at_5 | A1-A10 + B1-B10 + C1-C10 mean | tied or improved |
| Index latency (full poc.db, 349 symbols) | wall-clock seconds | < 10 min (current 0.6B is ~3-5 min) |
| Query latency (single embed call) | mean ms over 30 queries | < 200 ms |
| Peak RSS (CPU) or VRAM (GPU) | from process monitoring | < 16 GB (CPU swap headroom) |
| Install friction | minutes from clean checkout to first index | < 30 min (drop-in OR recipe doc) |

## Decision rule (encoded for runner agent)

```
LET candidates = [Qwen3-0.6B (control), Qwen3-8B, Harrier-27B (if compat), Voyage-code-3 (closed API control)]

FOR each candidate c:
  RUN B1-B7 + full-suite eval
  RECORD all 6 metrics

THEN apply ranked rule (highest priority first):

R1: IF any candidate has B1-B7 >= 70.9% AND latency < 10min AND license is Apache/MIT:
      decision = SWAP TO that candidate (highest B1-B7 wins ties)
      next: Phase 4.7 swap implementation SPEC

R2: ELIF Qwen3-8B has B1-B7 in [69.9%, 70.9%) (1-3 pp lift) AND VRAM available:
      decision = SOFT LIFT to 8B (worth doing but not urgent)
      next: queue Phase 4.7 swap behind higher-priority work

R3: ELIF no candidate beats 0.6B by >= 1 pp on B1-B7:
      decision = HOLD on 0.6B; document as "not-stale" with evidence
      next: amend ARCHITECTURE.md §9.8 to record re-evaluation date + thresholds for future re-trigger

R4: ELIF closed API (Voyage-code-3) wins by >= 5 pp:
      decision = INCONCLUSIVE; need open-weight follow-up search OR API-bridge architecture decision
      next: defer 1-2 quarters, re-check 2026-Q3 SOTA
```
